## Module Responsibility

`@orgtrace/scanner` handles all local SFDX filesystem scanning. It may use Node.js APIs (`fs`, `path`, `fast-glob`, `fast-xml-parser`) but must never import `vscode` or any UI code.

Two public functions:
- `discoverComponents(projectPath, query, typeFilter?)` — fast filename-based discovery for the picker UI. No file content reads for specialized types; reads content only for generic `-meta.xml` files to extract type.
- `scan(options)` — full dependency scan. Reads file contents and runs the parser registry against every matched file.

## Parser Registry

`fileParsers` is the single array of `FileParser` implementations. Each parser:
- `canParse(filePath)` — returns true if it handles this file extension/pattern
- `parse(context)` — returns `DependencyReference[]`

Current parsers (in order):
| File | Handles |
|------|---------|
| `apexParser.ts` | `.cls`, `.trigger` — line-by-line search, classifies relationship (Tests/Invokes/Extends/WritesField/References) |
| `flowParser.ts` | `.flow-meta.xml` — XML parse, recursive node search, emits one reference per match |
| `lwcJsParser.ts` | LWC `.js` files |
| `lwcHtmlParser.ts` | LWC `.html` files |
| `objectFieldParser.ts` | `.object-meta.xml`, `.field-meta.xml` |
| `permissionSetParser.ts` | `.permissionset-meta.xml` |
| `genericMetadataXmlParser.ts` | any `-meta.xml` not handled by specialized parsers — infers `relationshipType` from source type (integration types get `Configures`, field targets get `UsesField`, object targets get `UsesObject`) |

To add a new parser: create `src/parsers/myParser.ts`, add it to `fileParsers` in `src/parsers/index.ts`.

## Key Implementation Details

**Field name disambiguation** (`targetMatch.ts`): For `CustomField` targets with a qualified name (`Case.Field__c`), `targetSearchTerm()` strips the object prefix and searches by the bare field name. The emitted reference still carries the fully qualified `target.apiName`. This is critical for matching bare field references in Apex/Flow.

**Deduplication** (`referenceKey.ts`): References are keyed on `source|target|relationship|filePath|lineNumber|matchedText`. After scanning all files, duplicates are removed before splitting into `references` (inbound) vs `dependencies` (outbound).

**Inbound vs outbound split**: A reference where `ref.target.apiName === target.apiName` is inbound (something references our target). Otherwise it is outbound (our target references something else).

**Discovery rules** (`discoverComponents.ts`): `DISCOVERY_RULES` handles typed metadata by glob. Fields are qualified as `Object.Field__c`. Generic `-meta.xml` files are read to extract the root XML element as the type name.
